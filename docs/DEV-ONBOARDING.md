# Evenly — Developer Onboarding Guide

Welcome to Evenly. This doc gets you from zero to making changes in under 30 minutes.

---

## 1. Prerequisites

```bash
node --version    # 18+
npm --version     # 9+
java -version     # JDK 17 (for Android builds)
adb devices       # Android SDK installed
maestro --version # For E2E tests: brew install maestro
```

## 2. Local Setup

```bash
git clone https://github.com/ajayksingh/evenly.git
cd evenly

# Create .env from template
cp .env.example .env
# Fill in your Supabase URL + anon key (or use the team's shared .env)

npm install
npx expo start
# Press: w (web) | a (Android) | i (iOS)
```

## 5. Making Changes — By Screen

### To modify a screen:
1. Find the file in `src/screens/`
2. The screen receives `{ navigation, route }` props
3. Uses `useApp()` for global state and `useTheme()` for colors
4. Styles are memoized: `const styles = useMemo(() => getStyles(theme), [theme])`

### Key patterns used everywhere:
```js
// Theme-aware styling
const { theme, colorScheme } = useTheme();

// Global state
const { user, groups, friends, balances, refresh, currency } = useApp();

// Scroll-fade header
const scrollY = useRef(new Animated.Value(0)).current;
const headerBg = scrollY.interpolate({ inputRange: [0, 80], outputRange: [theme.headerBgTransparent, theme.headerBg] });

// Entrance animation (native only, with guard)
const animatedOnce = useRef(false);
useFocusEffect(useCallback(() => {
  if (!isWeb && !animatedOnce.current) {
    animatedOnce.current = true;
    screenOpacity.value = withTiming(1, { duration: 380 });
    screenTranslateY.value = withSpring(0, { damping: 18, stiffness: 120 });
  }
}, []));
```

### Screen-specific notes:

| Screen | File | Notes |
|--------|------|-------|
| HomeScreen | `HomeScreen.js` | Has cross-group settlements section, AdMob banner, skeleton state uses `lastLoadTimestamp` |
| GroupDetailScreen | `GroupDetailScreen.js` | Expenses use FlatList (virtualized). Has own realtime subscription. `memberCount` comes from useMemo. |
| AddExpenseScreen | `AddExpenseScreen.js` | Calculator uses safe parser (no eval/Function). 4 split modes. |
| SettleUpScreen | `SettleUpScreen.js` | useEffect for confetti MUST be before conditional returns (hooks rules). 3-step flow: form → processing → success. |
| ActivityScreen | `ActivityScreen.js` | Standalone tab — navigate to other stacks via `navigation.navigate('Home', { screen: ... })` |
| FriendsScreen | `FriendsScreen.js` | Friend cards are `<View>` not `<TouchableOpacity>` (no empty onPress). Single-pass partition in useMemo. |

## 6. Making Changes — By Feature Area

### Adding a new popup/alert
```js
import { themedAlert } from '../components/ThemedAlert';
themedAlert('Title', 'Message', 'success');  // types: success, error, warning, info
```
For confirmations:
```js
import { confirmAlert } from '../utils/alert';
confirmAlert({ title: 'Delete?', message: '...', confirmText: 'Delete', destructive: true, onConfirm: () => {} });
```
**Never use `Alert.alert()` directly.**

### Adding a new feature flag
1. Add default to `DEFAULTS` object in `src/services/flags.js`
2. Use in code: `getFlagSync('my_flag')` or `await getFlag('my_flag')`
3. To override remotely: insert row in Supabase `feature_flags` table

### Adding a new Supabase table
1. Write migration SQL: `supabase_migration_00N_description.sql`
2. Add RLS policies (SELECT, INSERT, UPDATE, DELETE as needed)
3. Run in Supabase Dashboard → SQL Editor
4. Add CRUD functions in `src/services/storage.js`
5. If realtime needed: `ALTER PUBLICATION supabase_realtime ADD TABLE tablename;`

### Adding a new screen
1. Create `src/screens/MyScreen.js`
2. Add to the appropriate Stack in `src/navigation/AppNavigator.js`
3. Lazy load if not a tab screen: `const MyScreen = lazy(() => import('../screens/MyScreen'));`
4. Use `useTheme()`, `useApp()`, scroll-fade header, entrance animation pattern

### Modifying the data model
1. Write a migration SQL file
2. Update `storage.js` CRUD functions
3. Update `AppContext.js` if new state needed
4. Add RLS policy in migration
5. Run migration in Supabase SQL Editor

## 7. State Management

Single memoized React Context (`AppContext`). No Redux.

### Data flow:
```
User action → storage.js function → Supabase/AsyncStorage
           → notifyWrite() → loadData(force=true) → setState
           → Realtime subscription → scheduleSync() → loadData()
```

### loadData phases:
```
Phase 1 (parallel): getGroups + getFriends + getGroupInvites + getFriendRequests
  → Set groups, friends, invites immediately (UI renders)

Phase 2 (parallel, needs groupIds): calculateBalances + getActivity
  → Set balances, activity (UI fills in)
```

### Stale-time gate:
`loadData()` skips if called within 5 seconds. Override with `loadData(true)`.

## 8. Supabase Setup

### New project setup:
1. Create project at [supabase.com](https://supabase.com)
2. Run SQL files in order:
   ```
   supabase_schema.sql
   supabase_migration_001.sql
   supabase_migration_002.sql
   supabase_migration_003.sql
   supabase_migration_004.sql
   supabase_migration_005.sql
   supabase_migration_006_rls.sql
   supabase_migration_007_expense_update.sql
   ```
3. Enable Google OAuth: Dashboard → Authentication → Providers → Google
4. Copy URL + anon key to `.env`

### RLS is enabled on ALL tables. Key rules:
- Users can read all profiles (needed for group member names)
- Users can only write their own data
- Group data scoped to creator + members (via JSONB membership check)
- `get_user_groups()` RPC uses SECURITY DEFINER (bypasses RLS)

### Free tier limits:
- 500 MB database
- 1 GB file storage
- 200 concurrent realtime connections
- 5 GB bandwidth/month
- **Projects pause after 7 days of inactivity** (keep-alive cron prevents this)

## 9. EAS Build Setup

### Profiles (eas.json):
| Profile | Output | Use |
|---------|--------|-----|
| `development` | Dev client | Local debugging with native modules |
| `preview` | APK | Internal testing on device |
| `production` | AAB | Play Store submission |

### Signing:
- Keystore: `android/app/evenly-release.jks` (EAS keystore)
- Credentials configured in `android/app/build.gradle` under `signingConfigs.release`
- **Never commit keystore passwords** to git (currently in build.gradle for local builds)

### Build commands:
```bash
# EAS cloud build (uses free tier quota)
eas build --platform android --profile production

# Local build (no quota needed)
npx expo prebuild --platform android --clean
cd android && ./gradlew assembleRelease     # APK
cd android && ./gradlew bundleRelease       # AAB for Play Store

# Install on device
adb install -r android/app/build/outputs/apk/release/app-release.apk
```

## 12. Conventions & Rules

| Rule | Details |
|------|---------|
| **No `Alert.alert()`** | Use `themedAlert()` or `confirmAlert()` |
| **No hardcoded colors** | Use `theme.*` tokens from ThemeContext |
| **No `console.log` in prod** | Gate behind `__DEV__` |
| **`keyboardShouldPersistTaps="handled"`** | On all scrollable forms |
| **`animatedOnce` guard** | On all entrance animations |
| **Optimistic updates** | For comments, reactions — revert on failure |
| **Feature flags** | Gate new/risky features via `flags.js` |
| **Progressive loading** | Set state after Phase 1, don't wait for Phase 2 |
| **No `eval()` or `Function()`** | Calculator uses safe parser |
